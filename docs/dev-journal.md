## 解决问题的思路和方法

### 项目启动流程
1. 先写设计文档（spec），明确需求和架构
2. 用 brainstorming skill 探索方案
3. 用 writing-plans skill 生成详细实现计划
4. 按计划逐步执行，配合 TDD

### 什么时候引入测试相关 Skill

**结论：按阶段引入，不要一开始就全装上。**

- **后端阶段（Task 1-9，Go）**：Go 自带测试框架 + testify 库就够，用 `superpowers:test-driven-development` skill 指导流程即可，无需额外 skill
- **前端阶段（Task 10+，React/TypeScript）**：React 组件测试更复杂，**在开始写 React 代码之前**安装 Webapp Testing Skill（Claude 官方 skill，需提前下载）

**原则：不要为了还没遇到的问题提前准备工具，用到时再引入。**

### 什么时候手动测试（点一点）

不要等全部做完再测，分三个节点验收：

| 节点 | 时机 | 测试方式 | 验证内容 |
|------|------|---------|---------|
| 节点1 | Task 9 后端完成后 | Postman 调接口 | 登录、账号、留言接口是否正常 |
| 节点2 | Task 13 登录页完成后 | 浏览器点击 | 登录流程是否跑通 |
| 节点3 | Task 18 全部完成后 | 完整真实流程 | 抓取→分类→后台展示全链路 |

**原则：发现问题越早，改起来代价越小。**

### 前端页面设计什么时候做

**不在后端阶段并行做，等 Task 9 完成后、Task 10 开始前做。**

原因：
- 设计文档已经定义了页面结构，够用
- 这是内部运营工具，不需要精心打磨视觉
- 后端接口没定下来前，设计出来的界面可能跟接口对不上，容易返工

做法：Task 9 后花半天画简单线框图（手画或简单工具），够指导开发就行，不用做精美设计稿。

## 学到的技巧

### 对接第三方未公开 API 时的策略（Task 12 经验）

微信公众号后台的接口没有文档，无法提前知道响应的确切字段名。这种情况下的做法：

1. **先拦截一切，再按启发式规则筛选。** 不要假设字段名，而是写一个"像不像评论数组"的判断函数，找到符合特征的数组（有 content、有名字或时间）就认为是目标数据。
2. **优先尝试已知常见键名，再做通用搜索。** 把 `elected_comment`、`comment_list` 等已知可能性排在前面，失败才做深度遍历，这样既能快速命中，又能应对未知格式。
3. **归一化层（normalise）独立出来。** 字段提取和格式转换放在一个专门函数里，业务逻辑不直接碰原始数据。以后即使接口字段变了，只改这一个函数。

**适用场景：** 任何需要对接无文档 API 或格式不稳定的数据源时，都可以用这套"宽进严出"的思路——入口宽泛拦截，出口按启发式提取，最后归一化成自己需要的格式。

### 在宿主页面注入代码时的安全原则

内容脚本（content script）运行在真实用户页面里，任何未捕获的错误都会影响用户体验：

- 所有拦截逻辑都包在 try/catch 里，出错静默忽略，保证原始请求正常返回。
- 整个脚本包在 IIFE（立即执行函数）里，避免污染全局作用域。
- 保存原始函数引用后再覆盖，确保 patch 可以调用回原始实现。

**适用场景：** 任何需要 monkey-patch 第三方页面（如 fetch/XHR/DOM 事件）的场景都适用这三条原则。
