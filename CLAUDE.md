# WeCatch

抓取微信公众号留言，通过大模型自动分类，筛选出有价值的留言供运营人员处理。

## 项目结构

- Chrome 插件（Manifest V3）：抓取留言 + React 管理后台页面
- Go 后端：REST API
- PostgreSQL：数据存储
- 大模型：通义千问（留言分类）

## 技术栈

- 前端：React（打包在 Chrome 插件内）
- 后端：Go
- 数据库：PostgreSQL
- 认证：JWT

## Chrome 插件调试经验

### 网络拦截失效时的标准排查流程

1. **先确认 content script 有没有注入**：在脚本顶部加 `console.log`，DevTools 必须在刷新页面之前打开，否则 `document_start` 阶段的日志会被错过
2. **确认 patch 是否存活**：用 `window.fetch.toString()` 检查当前 `window.fetch` 是不是自己的函数；如果被第三方页面脚本覆盖，拦截就是死路
3. **拦截失效立即换策略**：切换到主动请求——content script 直接调目标 API，天然带 Cookie，比拦截更简单可靠

### 对接未知 API 的方法

- 先在 Network 面板观察页面自己发了哪些请求，找到返回目标数据的那个
- 点开 Response 看真实数据结构，再写解析代码，不要假设字段名
- 在控制台直接 fetch 那个 URL 验证参数，把原始响应打印出来看
- **先探索，后编码**：不知道 action 名或参数叫什么时，先抓 Network 真实请求，不要猜；分页和游标逻辑尤其不能靠猜（`begin`、`max_id`、游标从哪个字段取都要从真实响应确认）
- 猜结构写出来的代码，改起来比从头写还慢；先花 20 分钟在 console 里探索清楚，比事后返工快得多
